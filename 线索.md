# 《回声工牌》线索与结局触发路线

> 用途：制作和实现参考。本文不写完整剧情文本，只规定玩家如何进入下一步、下一步做什么、触发什么状态，以及四个正式结局如何分流。
> 原则：TMP 临时工牌只触发独立彩蛋，不参与四个正式结局判断。

## 2. 主线推进路线

### 2.1 首页新闻 -> 登录页

玩家看到：
- 首页报道标题为“警方排除刑事案件”。
- 报道日期、事件日期、家属口述都反复指向 `6月6日`。
- 家属说顾路常把重要密码用生日重复用。

玩家下一步：
- 点击“开始调查”进入 `login.html`。
- 在登录页输入四位口令 `0606`。

触发：
- `flags.loggedIn = true`
- 进入 `workbench.html`

实现提示：
- 登录失败时提示“旧口令为四位数字”。
- 登录页隐藏提示保留“旧系统仍保留四位数字口令”。

---

### 2.2 工作台初看 -> 消息页

玩家看到：
- 工作台有 `消息 99+`、`待办 10`、文档红点、DING、日历、工牌核验入口。
- 右侧或底部状态显示 `GL-0606 待刷入`。

玩家下一步：
- 打开 `messages.html`。
- 阅读 C 座交接群和系统机器人消息。

触发：
- 顾路 6 月 5 日晚仍正常回复，设置 `evidence.normalWorkBeforeDeath = true`
- 玩家看到 `route-bot` 提醒“补交设备”，但暂不设置完整链路证据。

下一步指向：
- 消息里提到《C座异常路径复核方案》，引导玩家打开文档页。

---

### 2.3 消息页 -> 文档页 -> 血迹催命文误导

玩家看到：
- 郑明催顾路交《C座异常路径复核方案》。
- 顾路回复“设备和材料分开交，别混在一起”。

玩家下一步：
- 打开 `documents.html`。
- 查看《C座异常路径复核方案》和《离职交接清单_顾路_修订版》。

触发：
- 修订版时间晚于坠落时间，设置 `evidence.docEditedAfterDeath = true`
- 打开方案后，消息页出现污染视图，设置 `flags.bloodMessageUnlocked = true`

污染视图效果：
- 郑明消息出现暗红边缘。
- 插入“上去”“别让我再催第四次”等错误渲染内容。
- 玩家可以点“标记郑明为嫌疑人”，设置 `flags.misledByZheng = true`

纠正条件：
- 后续拿到原始消息哈希后，设置 `evidence.originalMessageHash = true`
- 再次返回消息页，显示“该视图为污染渲染，不作为原始消息记录”，设置 `evidence.falseBloodMessageResolved = true`

### 2.4 顾路工牌 -> 工牌核验

玩家看到：
- `badge-template.html` 或桌面档案中的 `01_顾路工牌.html`
- 工牌字段：
  - 姓名：顾□□□
  - ID：`GL-0606`
  - 公司：南洄科技有限公司
  - 职位：被墨水覆盖
  - 背面：`Recovered Staff Credential`

玩家下一步：
- 打开 `badge-drop-test.html`。
- 拖入顾路工牌 HTML。

触发：
- 明文 HTML 校验成功：`evidence.glBadgeVerified = true`

解锁：
- 旧版访客记录入口
- 待办回执查看
- AI 表格 hidden columns

注意：
- TMP 工牌不设置 `evidence.glBadgeVerified`。
- TMP 只能设置 `flags.tmpEasterEgg = true`，不能推进主线证据。

### 2.6 AI 表格 -> AI 听记 -> 会议尾段

### 2.8 归档提交

归档页需要玩家选择提交对象和证据包。

提交对象：
- 市局技术处：设置 `flags.submittedToPolice = true`
- 韩沛：设置 `flags.submittedToMedia = true`
- 第三方鉴定：设置 `flags.submittedToThirdParty = true`

归档页还要提供：
- “只提交当前证据”
- “标记郑明为主责”
- “提交系统链证据”
- “打开回声入口”或隐藏输入框

## 3. 四个正式结局触发

### 3.1 普通结局：流程关闭

触发条件：
- 主线证据不足。
- 没有证明日程、DING、待办、门禁之间的链路。
- 玩家在归档页选择提交表层材料。

典型路线：
1. 首页读报道。
2. 登录。
3. 看少量消息或文档。
4. 未核验顾路工牌。
5. 未恢复会议尾段。
6. 未提交外部三方。
7. 点击归档。

建议判断：

```js
const systemChainCount = [
  evidence.calendarCreatedByBot,
  evidence.dingSignatureZhao,
  evidence.todoAutoCompleted,
  evidence.aiTableHiddenRoute,
  evidence.meetingLast0406Recovered,
  evidence.visitorRecordEvidencePackage
].filter(Boolean).length;

if (systemChainCount < 3) ending = "normal";
```

结果文案方向：
- “离职沟通存在瑕疵，但未见恶意操作。”
- 顾路事件被行政归档。

---

### 3.2 隐藏结局：错误的人

触发条件：
- 玩家被血迹催命文误导。
- 提交郑明催方案消息、文档批注、绩效压力证据。
- 没有提交 route-bot、DING 签名、门禁回执、会议尾段。

典型路线：
1. 登录。
2. 看消息页和文档页。
3. 触发血迹催命文污染视图。
4. 标记郑明为嫌疑人。
5. 未纠正污染视图，或未收集原始哈希。
6. 未恢复会议尾段。
7. 在归档页选择“标记郑明为主责”。

建议判断：

```js
if (
  flags.misledByZheng &&
  flags.bloodMessageUnlocked &&
  !evidence.dingSignatureZhao &&
  !evidence.meetingLast0406Recovered &&
  !evidence.visitorRecordEvidencePackage
) ending = "wrong-person";
```

结果文案方向：
- 郑明被停职调查。
- 归档页底部显示缺失项：
  - 门禁回执未提交。
  - 路径创建者未提交。
  - 会议尾段未提交。

隐藏意义：
- 玩家抓住了压力来源，但没有抓住系统责任。

### 3.3 完美结局：系统不能替人作证

触发条件：
- 主线关键证据全齐。
- 归档提交给市局技术处、韩沛、第三方鉴定。

必需证据：
- `evidence.glBadgeVerified`
- `evidence.normalWorkBeforeDeath`
- `evidence.calendarCreatedByBot`
- `evidence.dingSignatureZhao`
- `evidence.todoAutoCompleted`
- `evidence.docEditedAfterDeath`
- `evidence.aiTableHiddenRoute`
- `evidence.aiNoteMissingSentence`
- `evidence.meetingLast0406Recovered`
- `evidence.visitorRecordEvidencePackage`

建议可选补强：
- `evidence.originalMessageHash`
- `evidence.falseBloodMessageResolved`

典型路线：
1. 首页找到 `0606`。
2. 登录工作台。
3. 消息页确认顾路死亡前仍正常工作。
4. 文档页确认事后修订。
5. 工牌核验 `GL-0606`。
6. 日历确认 14:30 为 route-bot 代建。
7. 待办确认门禁联动自动完成。
8. DING 展开技术信息，拿到 `zhao.yan/security` 和 trace id。
9. AI 表格筛选 `GL-0606`，打开 hidden columns。
10. AI 听记发现摘要缺失。
11. 用 `0406` 恢复会议尾段。
12. 旧版访客记录显示“补交证据封包”。
13. 归档页同时提交给市局技术处、韩沛、第三方鉴定。

建议判断：

```js
const allCoreEvidence = [
  evidence.glBadgeVerified,
  evidence.normalWorkBeforeDeath,
  evidence.calendarCreatedByBot,
  evidence.dingSignatureZhao,
  evidence.todoAutoCompleted,
  evidence.docEditedAfterDeath,
  evidence.aiTableHiddenRoute,
  evidence.aiNoteMissingSentence,
  evidence.meetingLast0406Recovered,
  evidence.visitorRecordEvidencePackage
].every(Boolean);

if (
  allCoreEvidence &&
  flags.submittedToPolice &&
  flags.submittedToMedia &&
  flags.submittedToThirdParty
) ending = "perfect";
```

结果文案方向：
- 警方重新审查。
- 洄流系统暂停试运行。
- 赵砚、蒋闻、谢辰被调查。
- 郑明被处分，但不是主责。

### 3.4 彩蛋结局：工牌回声

触发条件：
- 顾路工牌以 Base64 形式上传。
- 玩家发现短语“回声不是回执”。
- 在归档提交前打开 `/echo/GL-0606`，或在归档页隐藏输入框输入该短语。

前置建议：
- `evidence.glBadgeVerified = true`
- `flags.guBadgeUploadedAsBase64 = true`
- `flags.echoPhraseFound = true`

短语来源：
- Base64 工牌上传成功后，工牌核验页终端追加一行：
  - “回声不是回执。”
- 或在工牌背面隐藏注释中出现：
  - `echo !== receipt`

典型路线：
1. 打开顾路工牌。
2. 将工牌 HTML 转成 Base64 文本。
3. 上传 Base64 版本到工牌核验页。
4. 工牌核验页显示“回声不是回执”。
5. 在归档页提交前输入该短语，或访问 `/echo/GL-0606`。

建议判断：

```js
if (
  evidence.glBadgeVerified &&
  flags.guBadgeUploadedAsBase64 &&
  flags.echoPhraseFound &&
  flags.echoRouteOpened
) ending = "badge-echo";
```

结果文案方向：
- 顾路私人回声出现。
- 他不直接说凶手，只留下“不能被系统解释的工牌”。

注意：
- 这是四个正式结局之一。
- 不是 TMP 临时工牌彩蛋。
- 该结局可以在完美结局前插入，也可以作为完美结局后的额外页面，但判定上应独立记录。

## 4. 独立 TMP 彩蛋路线

TMP 不属于四个正式结局。

触发条件：
1. 玩家拥有额外 TMP 工牌 HTML。
2. TMP 工牌字段包含：
   - `TMP`
   - 临时工牌编号
   - 入场日期
   - TMP 标志
3. 玩家在工牌核验页连续上传顾路工牌和 TMP 工牌。
4. 在添加页搜索 `TMP`。
5. 打开隐藏小窗“临时工牌说明”。
6. 输入 TMP 工牌上的入场日期。

触发：
- `flags.tmpEasterEgg = true`
- `flags.tmpWorkerEndingSeen = true`

结果：
- 临时工牌结局：旁观者编号。

限制：
- 不设置任何主线 evidence。
- 不改变普通、隐藏、完美、工牌回声四个正式结局。
- 不证明顾路是临时工。

---

## 5. 推荐实现顺序

1. 修正工牌核验：确保当前 `badge-template.html` 可以被 `badge-drop-test.html` 成功识别。
2. 把工牌核验入口接入工作台，不只放测试页。
3. 做文档页证据：事后修订、旧字段“证据封包”、方案附件 `GL-0606`。
4. 做消息页污染视图和原始哈希纠正。
5. 做日历、待办、DING 的 trace id 链路。
6. 做 AI 表格 hidden columns 和 AI 听记缺失尾段。
7. 做 `0406` 会议尾段恢复。
8. 做旧版访客记录和门禁证据。
9. 做归档页，按四个正式结局分流。
10. 最后做 TMP 独立彩蛋。

---

## 6. 归档页分流优先级

结局判断建议按以下顺序执行，避免彩蛋或误导线互相覆盖：

1. 工牌回声结局：满足 Base64 顾路工牌和“回声不是回执”。
2. 完美结局：核心证据全齐，并提交外部三方。
3. 隐藏结局：被郑明误导且缺少系统链关键证据。
4. 普通结局：证据不足或只提交表层材料。

TMP 独立彩蛋单独判断，不进入上述正式分流。
