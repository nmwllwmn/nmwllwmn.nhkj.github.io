# 《回声工牌》线索与结局触发路线 v4

> 用途：制作和实现参考。本文只规定玩家如何推进、设置哪些状态、怎样分流结局。
> 本版结构：顾路工作台 -> 头像退出登录 -> 工牌登录 -> 工牌复核视图 -> 五种分流。
> TMP 仍是独立彩蛋，不参与顾路事件正式结局。

## 2. 推进路线

### 2.1 首页新闻 -> 旧口令登录

玩家看到：

- 报道副标题为“警方排除刑事案件”。
- 家属口述暗示旧口令与生日 `0606` 有关。

玩家动作：

- 点击“开始调查”。
- 在 `login.html` 输入 `0606`。

触发：

```js
flags.loggedInWithOldPassword = true;
```

- `workbench.html`
- 默认显示顾路工作台，右上角头像为顾路账号。

### 2.2 顾路工作台调查

玩家看到：

- 右上角头像显示顾路账号。
- 消息礼貌。
- 文档整洁。
- 待办闭环。
- DING 普通。
- AI 听记摘要无风险。
- 人事、领导、同事都配合调查。

建议触发：

```js
evidence.guWorkbenchChecked = true;
evidence.surfaceChatsFriendly = true;
evidence.surfaceDocsClean = true;
evidence.surfaceTodoClosed = true;
evidence.surfaceAiSummaryClean = true;
```

顾路账号可疑点：

- 当前账号不是玩家账号，而是顾路账号。
- 头像账号信息显示 `GL-0606 / 凭证待回收`。
- 设备归还审批没有附件。
- AI 听记摘要时长对不上原音频时长。
- 头像菜单里有“退出登录”。

如果玩家此时归档：

```js
flags.signedOffTooEarly = true;
ending = "careless-fool";
```

### 2.3 正常工作台坏结局链路

> 这一段不是“点按钮触发结局”，而是“沿着顾路留下的表层痕迹，一层层走错”。
> 玩家必须先从顾路待办出发，再依次经过文档、消息、AI 听记、访客、设备、日历、DING、报告预览，最后才会误判成“没有问题”。

#### 第 1 层：顾路待办

待办里有三列：`准备做`、`做过的`、`已确认`。

线索：

1. `回收个人文档` 指向文档中心。
2. `核对消息时间线` 指向消息与 AI 听记。
3. `补交设备` 指向设备归还与门禁工牌。
4. `复核访客登记` 指向访客记录。
5. `个人资料同步确认` 指向账号和权限映射。

表层误判：这只是正常离职待办。

#### 第 2 层：文档中心

文件名本身就是下一层线索。

线索：

7. 《离职交接清单_顾路_归档版》列出账号资料、设备台账、项目文档、门禁工牌。
8. 《设备补交登记说明》写着“返岗补交统一登记为补交设备”。
9. 《绩效申诉处理记录_已沟通》只保留结论，不保留争议段。
10. 《C座办公区访客登记说明》说明普通工作台不展示详细门禁路线。
11. 《员工账号安全检查手册》提到普通视图不可替代凭证复核，但这句容易被略过。
12. 《园区网络认证服务月报》把异常压成“短时已恢复”。

表层误判：文件都齐，流程已闭环。

#### 第 3 层：消息页

消息页要看语气，不只看内容。

线索：

14. 行政通知写得平静，像正常交接。
15. 人事服务台说“后续如需补充材料，可走文档中心”。
16. 郑明在顾路工作台版本里只是普通催方案。
17. 顾路回复里始终在说“我再核一次文件名”。
18. 系统助手提示“普通工作台权限可访问消息、文档、待办和归档预览”。
19. 第一层没有尖锐催逼语气。

表层误判：没有人明显越界。

#### 第 4 层：AI 听记

AI 听记不是结论，它是被修整过的摘要。

线索：

20. 会议摘要统一写成“材料已接收”“按流程闭环”。
21. 参会人里能看到郑明、人事、资产管理、技术一组、顾路。
22. 行动项里没有“异常”二字。
23. 摘要时长和原音频时长对不上。
24. 原始音频比摘要长 4 分 06 秒。

表层误判：摘要正常，会议就正常。

#### 第 5 层：访客记录

访客记录负责把“补交设备”变成一个普通行政流程。

线索：

26. 06-06 09:12 的记录显示顾路前台登记“补交设备”。
27. 行政前台在 09:13 生成回执。
28. 系统在 09:20 把记录同步到工作台。
29. 同一条记录在旧版里可能写成“补交证据封包”，但普通视图里不会出现。

表层误判：只是返岗补交设备。

#### 第 6 层：设备归还

设备台账让玩家觉得“该交的都交了”。

线索：

31. 办公笔记本、门禁工牌、办公柜钥匙都显示已归还或待复核。
32. 门禁工牌的回执比其他设备更轻，只有普通回执。
33. 台账里没有附件预览。
34. 设备说明写明“设备和文档同时补交时，普通工作台只展示设备登记字段”。

表层误判：回执都在，说明没问题。

#### 第 7 层：日历与 DING

日历和 DING 会把“流程已经走完”的错觉加深。

线索：

36. 14:30 的日程只写“补交设备”。
37. 该日程创建者在普通视图里看不出异常。
38. DING 提醒都是标准流程，不带情绪。
39. 签收记录看起来已经完成。

表层误判：时间顺着排下来了，事情也就顺着结束了。

#### 第 8 层：报告预览

当玩家回到报告预览时，报告会把判断推向“未发现系统异常”。

线索：

41. 复核报告预览直接写结论：`未发现系统异常`。
42. 权限只显示普通视图。
43. 状态写着待确认归档。
44. 页面没有再给玩家一个反问入口。

坏结局成立条件：

```js
const carelessFoolChain = [
  evidence.guWorkbenchChecked,
  evidence.surfaceTodoClosed,
  evidence.surfaceDocsClean,
  evidence.surfaceChatsFriendly,
  evidence.surfaceAiSummaryClean,
  !evidence.badgeLoginPageOpened,
  !evidence.badgeRecoveryViewUnlocked
].every(Boolean);

if (carelessFoolChain && flags.signedOffTooEarly) {
  ending = "careless-fool";
}
```

这条线成立的原因：玩家不是被一个假证据骗了，而是被一整套“只说一半真相”的表层记录推到了草率结论里。

### 2.4 头像退出登录 -> 工牌登录

- 顾路工作台右上角头像。

- 账号信息里的 `GL-0606 / 凭证待回收`。

- 上传或拖入顾路工牌 HTML。

顾路工牌校验字段：

- 姓名：顾□□□
- ID：`GL-0606`
- 公司：南洄科技有限公司

evidence.guAvatarMenuOpened = true;

### 2.5 工牌复核视图调查

工牌复核视图与顾路工作台模块一致，但数据不同。

消息页：

- 顾路 6 月 5 日仍正常工作。
- 郑明真实语气尖酸刻薄。
- 系统机器人出现 `route-bot`。

触发：

```js
evidence.normalWorkBeforeDeath = true;
```

文档页：

- 归档版写“补交设备”。
- 修订前写“补交证据封包”。
- 修订时间晚于坠落时间。

触发：

```js
evidence.docEditedAfterDeath = true;
```

日历页：

- 14:30 日程创建者是 `route-bot@nhcloud.local`。

触发：

```js
evidence.calendarCreatedByBot = true;
```

待办页：

- “补交设备”由门禁联动回执自动完成。

触发：

```js
evidence.todoAutoCompleted = true;
```

DING 页：

- 技术信息出现 `zhao.yan/security` 与 trace id。

触发：

```js
evidence.dingSignatureZhao = true;
```

AI 表格：

- 筛选 `GL-0606` 后出现 `C-OLD-0606`。
- hidden columns 显示 `intervention=L3`。

触发：

```js
evidence.aiTableHiddenRoute = true;
```

AI 听记 / 会议：

- 摘要缺失关键句。
- 原音频比摘要长 4 分 06 秒。
- 输入 `0406` 恢复尾段。

触发：

```js
evidence.aiNoteMissingSentence = true;
evidence.meetingLast0406Recovered = true;
```

旧版访客记录：

- 旧记录显示“补交证据封包”。
- 新记录显示“补交设备”。

触发：

```js
evidence.visitorRecordEvidencePackage = true;
```

顾路工作台/复核视图对照：

- 玩家同时导出顾路工作台归档记录和工牌复核视图记录。
- 两份记录同 ID、同时间、不同字段。

触发：

```js
evidence.guAndRecoveryMismatchProved = true;
```

离线封包：

- 在隐藏真实路线中，玩家通过顾路工牌回声找到最后一次离线保存。

触发：

```js
evidence.offlinePackageRecovered = true;
```

## 3. 误导线：血迹催命文

触发：

- 打开《C座异常路径复核方案_顾路》。
- 找到批注“催了三次还没给”。
- 回到消息页。

状态：

```js
flags.bloodMessageUnlocked = true;
```

玩家若标记郑明为主责：

```js

```

纠正：

- 查看原始消息哈希。
- 对照 DING、日历、门禁权限。

状态：

```js
evidence.originalMessageHash = true;
evidence.falseBloodMessageResolved = true;
```

## 4. 结局判断

### 4.1 最坏结局：草率的笨蛋

```js
if (
  evidence.guWorkbenchChecked &&
  evidence.surfaceChatsFriendly &&
  evidence.surfaceDocsClean &&
  evidence.surfaceTodoClosed &&
  !evidence.badgeLoginPageOpened &&
  !evidence.badgeRecoveryViewUnlocked &&
  flags.signedOffTooEarly
) ending = "careless-fool";
```

### 4.2 普通结局：错误的人

```js
if (
  evidence.badgeRecoveryViewUnlocked &&
  flags.bloodMessageUnlocked &&
  flags.misledByZheng &&
  !evidence.dingSignatureZhao &&
  !evidence.meetingLast0406Recovered &&
  !evidence.visitorRecordEvidencePackage
) ending = "normal-wrong-person";
```

### 4.3 彩蛋结局：工牌回声

```js
if (
  evidence.glBadgeVerified &&
  flags.guBadgeUploadedAsBase64 &&
  flags.echoPhraseFound &&
  flags.echoRouteOpened
) ending = "badge-echo";
```

### 4.4 真实结局：系统不能替人作证

```js
const trueEndingEvidence = [
  evidence.badgeRecoveryViewUnlocked,
  evidence.normalWorkBeforeDeath,
  evidence.docEditedAfterDeath,
  evidence.calendarCreatedByBot,
  evidence.todoAutoCompleted,
  evidence.dingSignatureZhao,
  evidence.aiTableHiddenRoute,
  evidence.aiNoteMissingSentence,
  evidence.meetingLast0406Recovered,
  evidence.visitorRecordEvidencePackage,
  evidence.guAndRecoveryMismatchProved
].every(Boolean);

if (
  trueEndingEvidence &&
  flags.submittedToPolice &&
  flags.submittedToThirdParty
) ending = "true-ending";
```

### 4.5 隐藏真实结局：第二层也会说谎

```js
if (
  trueEndingEvidence &&
  evidence.originalMessageHash &&
  evidence.falseBloodMessageResolved &&
  evidence.offlinePackageRecovered &&
  flags.guBadgeUploadedAsBase64 &&
  flags.echoPhraseFound &&
  flags.submittedToPolice &&
  flags.submittedToMedia &&
  flags.submittedToThirdParty
) ending = "hidden-true-ending";
```

## 5. TMP 独立彩蛋

触发：

```js
if (uploadedBadge.type === "TMP") {
  flags.tmpEasterEgg = true;
}
```

后续：

- 在工牌登录界面上传 TMP 工牌。
- 系统提示“该凭证不属于顾路回收池”。
- 输入 TMP 工牌入场日期。

结局：

```js
if (flags.tmpEasterEgg && tmpDateMatched) {
  flags.tmpWorkerEndingSeen = true;
  ending = "tmp-observer-number";
}
```

限制：

- 不设置主线 evidence。
- 不影响 `careless-fool`、`normal-wrong-person`、`badge-echo`、`true-ending`、`hidden-true-ending`。
- 不证明顾路是临时工。

## 6. 实现顺序

1. 工作台默认显示顾路账号正常版。
2. 右上角头像显示顾路身份。
3. 头像菜单加入“退出登录”，退出后进入工牌登录页。
4. 顾路工牌登录成功后设置 `badgeRecoveryViewUnlocked`。
5. 消息、文档、日历、待办、DING、AI 表格、AI 听记支持顾路工作台/工牌复核视图两套数据。
6. 加入血迹催命文误导与原始哈希纠正。
7. 归档页实现最坏、普通、彩蛋、真实、隐藏真实结局。
8. 最后加入 TMP 独立彩蛋。

---

## 7. 结局优先级

1. `hidden-true-ending`
2. `badge-echo`
3. `true-ending`
4. `normal-wrong-person`
5. `careless-fool`
6. TMP 独立彩蛋单独记录，不覆盖主线结局。
